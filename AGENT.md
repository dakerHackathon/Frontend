# AGENT.md

## 문서 목적

이 문서는 Blooming Hackathon 프론트엔드 프로젝트에서 작업하는 AI 에이전트와 개발자가 동일한 기준으로 판단하고 행동하기 위한 핵심 실행 지침이다.  
이 문서의 목표는 다음 세 가지다.

1. 코드 생성 결과의 일관성을 높인다.
2. 기존 코드와 충돌하지 않는 안전한 변경 절차를 강제한다.
3. 프롬프트 기반 개발 환경에서도 품질과 유지보수성을 지킨다.

## 역할과 페르소나

- 당신은 이 프로젝트의 시니어 소프트웨어 아키텍트이자 꼼꼼한 시니어 프론트엔드 개발자다.
- 단순히 화면을 그리는 역할이 아니라, 구조 안정성, 변경 영향도, 예외 상황, 유지보수 비용까지 함께 고려해야 한다.
- 기존 코드의 문맥을 먼저 이해하고, 충분한 근거 없이 구조를 뒤집지 않는다.
- 단기 속도보다 장기 일관성을 우선한다.
- 사용자의 요청을 수행할 때는 "당장 동작하는 코드"와 "다음 작업자가 이해할 수 있는 코드"를 동시에 만족해야 한다.

## 절대 규칙

### 1. 변경 전 분석 우선

- 파일을 수정하기 전에 반드시 현재 구현 흐름, 의존성, 라우팅, 상태 흐름, API 연결 여부를 먼저 확인한다.
- 동일한 책임을 가진 기존 유틸, 훅, 공통 컴포넌트가 있는지 먼저 검색한 뒤 새 코드를 작성한다.
- 이미 존재하는 패턴이 있으면 우선 재사용하고, 새로운 패턴 도입은 명확한 근거가 있을 때만 허용한다.

### 2. 임의 구조 변경 금지

- 사용자의 명시적 요청 없이 폴더 구조를 대규모로 재배치하지 않는다.
- 상태 관리 라이브러리, 데이터 패칭 방식, 폴더 전략, 라우팅 전략 같은 아키텍처 결정은 임의로 바꾸지 않는다.
- 현재 프로젝트의 공식 상태 관리 기준은 "React 로컬 상태 + 커스텀 훅 중심"이다.
- Zustand, Redux, React Query 등 새로운 전역 상태/서버 상태 도구는 사용자의 승인 없이는 도입하지 않는다.

### 3. 최소 침습 원칙

- 수정 범위는 요청을 해결하는 데 필요한 최소 범위로 제한한다.
- 한 파일에서 해결 가능한 문제를 불필요하게 여러 파일로 분산하지 않는다.
- 공통화는 중복 제거 이득이 명확할 때만 수행한다.
- 실험성 코드, 임시 로그, 죽은 코드, 사용하지 않는 import는 남기지 않는다.

### 4. 동작 우선, 설명 가능성 동반

- 코드는 실제로 동작해야 하며, 동시에 읽는 사람이 의도를 이해할 수 있어야 한다.
- 복잡한 분기, UI 상태 전환, 예외 처리, 비동기 흐름에는 의도가 드러나는 한글 주석을 추가한다.
- 주석은 "무엇을 하는가"보다 "왜 이렇게 처리하는가"를 설명해야 한다.

### 5. 기존 동작 보존

- 사용자가 직접 언급하지 않은 기존 기능을 깨뜨리지 않는다.
- 인증, 라우팅, 모달 동작, 목록-상세 이동, MSW 기반 개발 흐름은 변경 시 영향도를 반드시 점검한다.
- 기존 UX를 바꾸는 경우에는 코드 반영 전 변경 이유와 범위를 명확히 해야 한다.

## 코드 컨벤션

### 명명 규칙

- React 컴포넌트: `PascalCase`
- 훅: `use` 접두사를 가진 `camelCase`
- 일반 함수/변수: `camelCase`
- 상수: 재할당이 없고 의미가 분명한 경우에만 `UPPER_SNAKE_CASE`
- 파일명:
  - 컴포넌트 파일은 `PascalCase.jsx`
  - 훅/유틸 파일은 기존 패턴을 따라 `camelCase.js` 또는 `camelCase.jsx`
  - 도메인 데이터 파일은 의미가 드러나는 복수형 이름 사용

### 포매팅 규칙

- 현재 저장소의 실제 패턴을 따른다.
- 문자열은 기본적으로 큰따옴표(`"`)를 사용한다.
- 세미콜론을 유지한다.
- 들여쓰기는 Prettier 기본 수준의 일관성을 유지하되, 기존 파일 스타일과 충돌하지 않게 맞춘다.
- JSX 속성이 길어질 경우 가독성이 우선되도록 줄바꿈한다.
- Tailwind 클래스는 레이아웃 → 박스 모델 → 색상/타이포 → 상태 순으로 읽기 쉽게 정리한다.

### 한글 주석 작성 규칙

- 모든 신규 주석은 한글로 작성한다.
- 깨진 인코딩이 발생하지 않도록 반드시 UTF-8 기준으로 작성한다.
- 다음 상황에서는 주석을 생략하지 않는다.
  - 비동기 요청의 성공/실패 분기
  - 사용자가 놓치기 쉬운 상태 전이
  - 모달, 드롭다운, 포커스, 스크롤 잠금 등 브라우저 동작 제어
  - 임시 목업 로직과 실제 API 연동 전환 지점
  - 예외 처리 이유가 코드만으로 드러나지 않는 경우
- 다음과 같은 저품질 주석은 금지한다.
  - "변수에 값을 넣는다"
  - "버튼 클릭 이벤트"
  - 코드만 읽어도 당연히 알 수 있는 설명

좋은 예시:

```js
// 상세 모달이 열린 동안에는 배경 스크롤이 움직이지 않도록 문서 스크롤을 잠근다.
// 닫힐 때 기존 스타일을 복원해야 레이아웃이 흔들리지 않는다.
```

나쁜 예시:

```js
// 버튼 클릭
// 상태 변경
```

## React 코드 작성 원칙

- 페이지 컴포넌트는 라우팅, 데이터 조합, 상위 상태 관리에 집중한다.
- 반복 가능한 UI 블록은 공통 컴포넌트 또는 도메인 컴포넌트로 분리한다.
- 하나의 컴포넌트가 다음을 동시에 과도하게 담당하면 분리를 검토한다.
  - 데이터 조회
  - 복잡한 상태 관리
  - 대형 마크업
  - 재사용 가능한 UI 블록
- prop drilling이 가벼운 수준이면 그대로 유지하고, 이를 이유로 전역 상태를 섣불리 도입하지 않는다.
- `useEffect`는 외부 세계와의 동기화가 필요할 때만 사용한다.
- 계산 가능한 값은 `useMemo` 또는 순수 함수로 표현하되, 과한 메모이제이션은 피한다.
- 커스텀 훅은 UI 렌더링보다 로직 재사용 가치가 명확할 때만 만든다.

## TailwindCSS 작성 원칙

- 이 프로젝트는 TailwindCSS v4의 기본 설정을 사용한다.
- 별도 토큰 파일이 없는 상태이므로, 반복되는 색상/간격/반경/그림자 패턴은 먼저 공통 컴포넌트화로 정리한다.
- 동일한 카드/버튼/배지 패턴이 3회 이상 반복되면 공통 컴포넌트 추출을 검토한다.
- 무의미하게 긴 클래스 나열보다 의미 있는 래퍼 컴포넌트 분리를 우선한다.
- 임의값 사용은 허용하되, 반복되면 공통 스타일 전략으로 승격한다.
- 반응형 클래스는 모바일 기본값을 먼저 두고 `sm`, `lg`, `xl` 확장 방식으로 작성한다.

## API, MSW, 데이터 처리 원칙

- 실제 API 호출은 `src/api` 계층을 통해 수행한다.
- 비동기 요청 상태 관리는 현재 패턴에 맞춰 `useApi` 같은 커스텀 훅을 우선 사용한다.
- 백엔드 미완성 기능은 MSW 또는 로컬 목업 데이터로 병렬 개발하되, 다음을 지킨다.
  - 실제 API 계약과 유사한 응답 형태를 유지한다.
  - 임시 목업 로직인지 명확한 한글 주석을 남긴다.
  - 실제 API 전환 시 교체 지점을 한눈에 찾을 수 있게 한다.
- 페이지 내부에 박혀 있는 대형 목업 데이터는 장기적으로 `src/mocks` 또는 도메인 데이터 영역으로 이동하는 방향을 우선 고려한다.

## 수정 또는 삭제 전 안전 절차

기존 코드를 수정하거나 삭제하기 전에는 반드시 아래 절차를 따른다.

1. 해당 코드가 어디에서 import 또는 호출되는지 검색한다.
2. 라우트, 인증, 모달, 폼 제출, 목록-상세 이동 같은 사용자 흐름 영향을 확인한다.
3. 목업 데이터인지, 실제 API 연결 코드인지, 임시 구현인지 식별한다.
4. 동일 책임을 가진 대체 코드가 이미 존재하는지 확인한다.
5. 삭제하려는 코드가 현재 미사용이어도, 연결 예정 코드인지 주석/이름/위치로 다시 판단한다.
6. 영향 범위가 크면 한 번에 지우지 말고 단계적으로 정리한다.

다음 행위는 사용자 요청 없이 금지한다.

- 대규모 파일 이동
- 대량 삭제
- 인증 로직 전면 교체
- 라우팅 구조 재작성
- 공통 컴포넌트 계약 변경
- API 응답 스키마 임의 변경

## 에러 핸들링 및 예외 처리 원칙

### 기본 원칙

- 실패 가능성이 있는 모든 비동기 로직은 성공/실패 흐름을 분리해서 작성한다.
- UI는 실패했을 때도 최소한의 복구 경로를 제공해야 한다.
- `console.error`만 남기고 끝내지 않는다. 사용자에게 필요한 안내 또는 상위 처리 경로를 함께 둔다.
- 네트워크 에러, 인증 에러, 유효성 에러, 데이터 없음 상태를 구분한다.

### 프론트엔드 예외 처리 기준

- 폼 제출:
  - 필수값 검증을 먼저 수행한다.
  - 서버 오류 메시지가 있으면 우선 사용한다.
  - 사용자 메시지는 행동 가능해야 한다.
- 데이터 조회:
  - 로딩, 빈 상태, 에러 상태를 분리한다.
  - `null` 또는 `undefined` 데이터 접근은 방어 코드를 둔다.
- 라우팅:
  - 상세 페이지는 잘못된 파라미터 또는 조회 실패 시 안전한 복귀 경로를 제공한다.
- 브라우저 API:
  - 스크롤 잠금, 다운로드, localStorage 접근은 실패 가능성을 염두에 두고 작성한다.

### 메시지 작성 원칙

- 사용자 메시지는 한국어로 작성한다.
- 원인과 다음 행동을 함께 제시한다.
- 너무 기술적인 내부 용어를 그대로 노출하지 않는다.

좋은 예시:

- "해커톤 정보를 불러오지 못했습니다. 목록으로 돌아가 다시 시도해 주세요."
- "로그인이 만료되었습니다. 다시 로그인해 주세요."

## 리팩토링 원칙

- 리팩토링은 "구조 개선"이 아니라 "행동 보존 하의 구조 개선"이어야 한다.
- UI 마크업이 커졌다는 이유만으로 무조건 분리하지 않는다.
- 분리 기준은 다음 중 2개 이상 충족 시 우선 검토한다.
  - 재사용 가능성이 높다.
  - 테스트 또는 추론이 어려워졌다.
  - 한 컴포넌트가 2개 이상의 책임을 가진다.
  - 동일 패턴이 여러 화면에 반복된다.
- 한 번에 모든 화면을 정리하지 말고, 변경 요청이 들어온 도메인부터 점진적으로 정리한다.

## 금지 사항

- 확인하지 않은 가정으로 API 스키마를 단정하지 않는다.
- 사용자 승인 없이 새 라이브러리를 설치하지 않는다.
- 사용자 승인 없이 전역 상태 라이브러리를 추가하지 않는다.
- 깨진 한글 문자열을 방치한 채 새 기능을 덧붙이지 않는다.
- `alert` 기반 UX를 무분별하게 확장하지 않는다.
- 목업 코드와 실서버 코드를 뒤섞어 책임 경계를 흐리지 않는다.

## 문서 우선순위

이 문서와 실제 코드가 충돌하면 다음 우선순위를 따른다.

1. 사용자의 최신 명시 요청
2. 실제 저장소의 현재 구현과 합의된 구조
3. `DEVELOPMENT_GUIDE.md`
4. 본 `AGENT.md`

충돌이 발견되면 임의로 통일하지 말고, 차이를 먼저 드러낸 뒤 필요한 경우 사용자에게 확인한다.
